# Style Guide

## Purpose

This style guide exists to keep the product coherent across:

- interface writing
- lesson writing
- visual design
- metadata
- new features
- future course additions

It should help the site feel like one product with one point of view.

## Brand Voice

The site voice should feel:

- calm
- intelligent
- premium
- supportive
- clear
- serious without being stiff

It should not feel:

- corporate
- institutional
- aggressively playful
- hype-heavy
- academic in a deadening way

## Interface Writing

UI labels should be:

- short
- concrete
- human
- confident

Prefer:

- “Continue learning”
- “Open syllabus”
- “Time remaining”
- “High-trust sources”

Avoid:

- vague dashboard jargon
- technical labels users should not have to decode
- overly promotional copy on utility surfaces

## Lesson Writing Voice

Lesson text should generally be:

- professional
- engaging
- accessible
- high-signal

When writing for beginner learners:

- explain terms cleanly
- keep the tone friendly
- avoid condescension
- use examples generously

When writing for more advanced learners:

- keep depth high
- keep clarity high
- avoid showing off

## Typography Guidance

Typography should create:

- strong page openers
- elegant hierarchy
- calm reading rhythm
- visible but restrained metadata

Guidelines:

- large headlines may feel almost editorial
- section labels should be crisp and controlled
- body copy should breathe
- long content should not feel cramped

## Surface and Layout Guidance

Use cards and panels intentionally, not automatically.

Preferred qualities:

- rounded
- light
- soft-shadowed
- well-spaced
- premium

Avoid:

- dense dashboard clutter
- rigid enterprise grid feel
- overuse of small boxes for everything

## Color Guidance

Color should serve:

- hierarchy
- emphasis
- state
- atmosphere

Color should not:

- dominate the product
- make the UI noisy
- reduce readability

Accent use should feel strategic rather than constant.

## State Labels

Current state language:

- Not Started
- Started
- In Progress
- Completed

Use these consistently.

Do not invent alternate state vocabularies casually.

## Course Metadata Standards

Course metadata should remain consistent across catalog and course pages.

Preferred metadata categories:

- category
- difficulty
- guided time
- lesson count
- module count
- progress

Metadata should support decision-making, not overwhelm the page.

## Resource Standards

Resource descriptions should:

- explain why the resource matters
- signal trust or authority
- connect the resource to the learning objective

They should not:

- read like generic link blurbs
- overclaim
- sound marketing-driven

## Feature Design Standards

When adding new product features, ask:

1. Does this make learning clearer, deeper, or more motivating?
2. Does this preserve the premium calmness of the interface?
3. Does this fit the static-site architecture cleanly?
4. Does this make the platform feel more like a product and less like a pile of widgets?

If the answer is mostly no, the feature should be reconsidered.

## Flat-Site Taste Standard

The project should always feel intentionally built for the flat-site medium.

That means:

- no faux-backend theater
- no fake complexity for prestige
- no features that pretend to be smarter than they are

Instead, the site should feel:

- elegant
- transparent
- responsive
- surprisingly capable

## One-Sentence Standard

Every new piece of writing, design, or logic should make the site feel more like a premium self-education platform and less like a static project demo.
